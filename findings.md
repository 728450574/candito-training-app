## 一、Candito 6周力量计划分析

### 1.1 计划来源
- **原作者**: Johnnie Candito (知名力量举运动员/教练)
- **官方网站**: http://www.canditotraininghq.com
- **中文翻译**: 滕达(@力量营)
- **计划类型**: 力量举专项训练 (深蹲/卧推/硬拉)

### 1.2 训练周期结构

| 参数 | 详情 |
|------|------|
| 总周数 | 6周 |
| 训练频率 | 每周4-5天 |
| 周期类型 | 累积-强化-峰值 |

#### 分周详情

**第1周 - 肌肉调理 (中等难度)**
- 深蹲: 4组×6次 @ 80% 1RM
- 硬拉: 2组×6次 @ 75% 1RM
- 卧推: 金字塔组，最高6次 @ 70% 1RM
- 辅助训练: 标准容量

**第2周 - 肌肉调理/增肌 (高难度)**
- 引入 MR10 (Max Reps 10) 极限次数组
- 深蹲加量组: 5组×3次
- 深蹲减量组: 根据MR10表现调整
- 硬拉改为变式动作

**第3周 - 线性最大超负荷**
- 主项: 3组×4-6次
- 辅助项: 3组×6次
- 无可选项目，专注主项

**第4周 - 适应大重量**
- 深蹲: 3组×3次，渐进增重
- 卧推: 3次组，尝试1-2次极限
- 恢复辅助训练容量

**第5周 - 高强度力量训练**
- 深蹲: 1-4次 @ 97.5% 1RM
- 硬拉: 1-4次 @ 92.5% 1RM
- 卧推: 1-4次 @ 87.5% 1RM
- 减量硬拉辅助

**第6周 - 测试/减载/跳过**
三个选项:
1. 跳过，用第5周数据预估新1RM
2. 减载周 (重做第1周但降低容量)
3. 实测1RM

### 1.3 重量计算公式

基于用户1RM百分比:

```javascript
// 主要训练重量 (% of 1RM)
const weekWeights = {
  week1: { squat: 0.80, bench: 0.70, deadlift: 0.75 },
  week2: { squat: 0.80, bench: 0.65, deadlift: 0.80 }, // MR10组
  week3: { squat: 0.875, bench: 0.775, deadlift: 0.825 },
  week4: { squat: 0.90, bench: 0.80, deadlift: 0.875 },
  week5: { squat: 0.975, bench: 0.875, deadlift: 0.925 }
};

// 预估新1RM (基于第5周完成情况)
// 完成2次 ×1.03, 3次 ×1.06, 4次 ×1.09
function estimateNew1RM(weight, reps) {
  const multipliers = { 2: 1.03, 3: 1.06, 4: 1.09 };
  return weight * (multipliers[reps] || 1);
}
```

## 三、关键概念：周期管理

### 3.1 为什么需要周期管理？

Candito不是一次性计划，而是**可循环的训练体系**：

```
周期1 (6周)
    ↓ 第6周决策
    ├─ 开启新周期 → 周期2 (使用预估1RM)
    ├─ 1RM测试周 → 实测新1RM → 开启新周期
    └─ 减载周 → 减载后 → 开启新周期
```

### 3.2 周期状态流转

```
[创建周期] → [进行中：第1-5周] → [第6周决策] → [完成]
                                          ↓
                        ┌─────────────────┼─────────────────┐
                        ↓                 ↓                 ↓
                   [新周期]          [1RM测试]          [减载]
                        │                 │                 │
                        └─────────────────┴─────────────────┘
                                          ↓
                                   [创建新周期]
```

### 3.3 暂停功能设计

#### 为什么需要暂停？
- **小长假**（如国庆、春节）：5-7天无法训练
- **出差/旅行**：临时外出无法去健身房
- **短期身体不适**：感冒、轻微受伤等

#### 暂停场景示例

```
正常训练流程：
Day1(周一) → Day2(周三) → Day3(周五) → Day4(下周一)

暂停场景1 - 顺延（推荐）：
Day1(周一) → [暂停: 周三-周日出去玩] → Day2(下周一) → Day3(下周三)...
                                        ↑
                                   日期自动顺延5天

暂停场景2 - 跳过：
Day1(周一) → [暂停跳过Day2] → Day3(周五实际训练) → Day4(下周二)...
                               ↑
                          保持原计划，但日期变为实际训练日
```

#### 暂停状态流转

```
[训练中] → [点击暂停] → [输入暂停信息] → [暂停中]
                                              ↓
[恢复训练] ← [选择恢复方式] ← [点击恢复] ←─┘
   ↓
根据选择：
  - 顺延：暂停天数加到后续所有训练日期
  - 跳过：保持训练计划，从恢复日期继续
```

#### 多次暂停支持

一个周期可以有多个暂停记录：

```
周期1
├── Week1 Day1-3: 正常训练
├── [暂停1: 春节7天] → 顺延
├── Week1 Day4 - Week3: 正常训练
├── [暂停2: 出差3天] → 跳过
└── ...
```

### 3.4 从任意位置重新开始

#### 为什么需要？
- **假期后状态下滑**：想从Week1重新开始
- **某周没完成好**：想重做Week3
- **受伤恢复**：从Week2重新开始，降低重量

#### 重新开始选项

**选项1：从任意位置开始**
```
已完成：Week1 Day1-4 (3/1-3/10)
用户选择：从Week2 Day1开始
选择日期：3/15 (用户指定实际日期)

结果时间线：
Week1 Day1-4 (3/1-3/10) [已完成，保留]
         ↓
[重新开始] Week2 Day1 (3/15) → Week2 Day2 (3/17) → ...
                     ↑
              用户选择的实际日期
```

**选项2：调整1RM后重新开始**
```
用户发现原1RM过高，想从Week1重新开始
- 原1RM：深蹲100kg
- 新1RM：深蹲90kg
- 选择日期：3/20

结果：按新1RM重新计算所有重量，从Week1 Day1开始
```

#### 重新开始 vs 暂停的区别

| 功能 | 暂停 | 重新开始 |
|------|------|---------|
| 触发时机 | 当前正在进行的周期 | 任意已完成位置 |
| 保留历史 | ✅ 保留 | ✅ 保留（形成分支） |
| 日期变化 | 自动顺延/跳过 | 用户选择新日期 |
| 1RM调整 | ❌ 不变 | ✅ 可选调整 |
| 次数 | 多次 | 多次 |

### 3.5 日期管理核心规则

#### 核心原则
**所有训练日期必须与实际日历日期一致**，系统需要准确记录：
1. **原始计划日期**（基于开始日期+训练频率，永远不变）
2. **当前日程日期**（考虑暂停顺延后，用于显示"今天该做什么"）
3. **实际训练日期**（用户实际打卡的日期，必须与实际日历一致）

#### 日期字段设计

```typescript
interface TrainingDay {
  // 位置标识（不变）
  weekNumber: number;
  dayNumber: number;
  dayOffset: number;             // 相对于开始日期的偏移天数

  // 日期字段
  originalDate: string;          // 原始计划日期 = startDate + dayOffset
  scheduledDate: string;         // 当前日程日期（考虑暂停顺延）

  // 实际完成记录
  completedDate?: string;        // 实际完成的日期（打卡时填写）
}
```

#### 日期计算示例

**场景1：正常训练**
```
开始日期：3/1
Day1: originalDate=3/1, scheduledDate=3/1, completedDate=3/1
Day2: originalDate=3/3, scheduledDate=3/3, completedDate=3/3
```

**场景2：暂停后顺延**
```
开始日期：3/1
暂停：3/3-3/7（5天）

Day1: originalDate=3/1, scheduledDate=3/1, completedDate=3/1
Day2: originalDate=3/3, scheduledDate=3/8(+5), completedDate=3/8
Day3: originalDate=3/5, scheduledDate=3/10(+5), completedDate=3/10
```

**场景3：重新开始**
```
已完成Week1 (3/1-3/10)
重新开始：从Week2 Day1，选择日期3/20

Week1: originalDate=3/1-3/10, completedDate=3/1-3/10 [历史保留]
Week2 Day1: originalDate=3/12, scheduledDate=3/20, completedDate=3/20
Week2 Day2: originalDate=3/14, scheduledDate=3/22, completedDate=3/22
```

#### 关键保证

1. **实际日期永远正确**：用户打卡时记录的是真实日期
2. **历史记录可追溯**：originalDate保留原始计划
3. **当前日程清晰**：scheduledDate显示"今天该做什么"
4. **支持多种场景**：暂停、重新开始都不会导致日期混乱

#### 多次暂停支持

一个周期可以有多个暂停记录：

```
周期1
├── Week1 Day1-3: 正常训练
├── [暂停1: 春节 7天] → 顺延
├── Week1 Day4 - Week3: 正常训练
├── [暂停2: 出差 3天] → 跳过
└── ...
```

### 3.6 错过训练自动检测与处理

#### 问题场景

用户突然出差，**忘记手动暂停**，5天后回来发现：
- 错过了3个训练日
- 首页显示"今天该做Week2 Day3"（但已经是5天后）
- 计划与实际日期严重错乱

#### 为什么不自动暂停？

自动暂停的问题：
- 不知道用户是想**补练**还是**跳过**
- 可能用户已经自己在别处完成了训练
- 自动操作可能导致数据混乱

**更好的方案：检测 + 批量处理**

#### 检测机制

```typescript
// 每次打开应用时检测
function checkMissedWorkouts(): MissedWorkout[] {
  const today = getToday();
  const scheduledDays = getScheduledDaysBefore(today);

  return scheduledDays.filter(day => {
    // 计划日期已过
    const isPast = day.scheduledDate < today;
    // 未打卡
    const notCompleted = !day.completedDate;
    // 未标记跳过
    const notSkipped = !day.isSkipped;

    return isPast && notCompleted && notSkipped;
  });
}
```

#### 处理流程

```
[打开应用]
   ↓
[检测是否有错过的训练日]
   ↓ 有
[弹出"处理错过的训练"界面]
   ↓
[显示错过的训练日列表]
   ↓
[用户批量选择处理方式]
   ↓
[应用处理，调整后续日期]
```

#### 处理方式

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **补练** | 按原日期补打卡 | 出差回来健身房补练 |
| **跳过** | 标记为跳过 | 确实没条件训练 |
| **顺延** | 将训练移到今天之后 | 想补但想按顺序做 |
| **出差跳过** | 批量标记某日期范围为跳过 | 已知出差日期范围 |

#### 具体场景示例

**场景1：出差3天，回来后补练**
```
情况：错过了Day2(3/3)、Day3(3/5)、Day4(3/8)
今天是：3/10

用户选择：
- Day2: 补练（completedDate=3/3）
- Day3: 补练（completedDate=3/5）
- Day4: 顺延到3/10之后

结果：
Day2(3/3) ✓, Day3(3/5) ✓, Day4(3/11顺延), Day5(3/13)...
```

**场景2：出差5天，不想补练**
```
情况：错过了Day2-Day4

用户选择：出差跳过（3/3-3/7）

结果：
- Day2-Day4标记为跳过
- Day5自动顺延到3/10
- 后续日期相应顺延
```

### 3.7 终止功能设计

#### 为什么需要终止？
- **长期受伤**：需要数月恢复，无法继续当前计划
- **计划不适合**：发现计划难度过大/过小，想换计划
- **失去动力**：长期不想训练，暂时放弃

#### 终止 vs 完成的区别

| 状态 | 含义 | 数据保留 | 可恢复 |
|------|------|---------|--------|
| 已完成 | 正常走完6周 | ✅ 全部保留 | ❌ 不可恢复 |
| 已终止 | 提前结束 | ✅ 保留已完成部分 | ❌ 不可恢复（可查看） |

#### 终止后的数据查看

已终止的周期可以：
- 查看已完成的训练记录
- 导出数据
- 作为历史参考
- **不可修改**（只读模式）

### 3.8 数据保留策略

每个周期保存：
- **计划数据**: 6周完整计划
- **实际训练数据**: 每次训练的重量、次数、完成情况
- **训练笔记**: 每次训练的感受记录
- **体测数据**: 周期开始/结束的体重、围度等

```
[选择导入文件]
   ↓
[读取并解析JSON]
   ↓
[数据校验]
   ├── 格式校验（schema验证）
   ├── 完整性校验（checksum）
   └── 版本兼容性检查
   ↓
[冲突检测]
   ├── 是否有重复周期？
   ├── 是否有重复记录？
   └── 显示差异对比
   ↓
[用户确认]
   ├── 合并（智能合并重复项）
   ├── 覆盖（删除现有，使用导入）
   └── 取消
   ↓
[数据导入]
   ↓
[完成提示]
```

#### 冲突处理策略

| 冲突类型 | 处理方式 |
|----------|---------|
| 完全相同的周期ID | 提示用户：跳过/覆盖/保留两者 |
| 相同日期不同内容 | 显示对比，让用户选择保留哪个 |
| 导入数据损坏 | 提示错误，拒绝导入 |
| 版本不兼容 | 提示升级应用或数据太新 |

#### H5端导入实现

```typescript
// 文件选择
const input = document.createElement('input');
input.type = 'file';
input.accept = '.json';
input.onchange = (e) => {
  const file = e.target.files[0];
  const reader = new FileReader();
  reader.onload = (event) => {
    const data = JSON.parse(event.target.result);
    importData(data);
  };
  reader.readAsText(file);
};

// 粘贴导入（方便）
const importFromClipboard = async () => {
  const text = await navigator.clipboard.readText();
  const data = JSON.parse(text);
  return importData(data);
};
```

#### 小程序端导入实现

```typescript
// 微信：从聊天记录选择文件
wx.chooseMessageFile({
  count: 1,
  type: 'file',
  success: (res) => {
    const file = res.tempFiles[0];
    // 读取文件内容
    const fs = wx.getFileSystemManager();
    fs.readFile({
      filePath: file.path,
      encoding: 'utf8',
      success: (data) => {
        importData(JSON.parse(data.data));
      }
    });
  }
});
```

**⚠️ 第2周特殊逻辑 - MR10和加减量组**

**MR10 (Max Reps 10)**: 极限次数组，但不超过10次

**深蹲加量组 (Day 1)**:
- 在MR10组完成后
- 加5磅或2.5kg
- 做5组×3次
- 组间休息60秒
- 如果MR10组无法完成最低要求的8次，给最大重量减轻2.5%然后继续
- 不管能否完成MR10组最低要求的8次，都需做5组3次的训练

**深蹲减量组 (Day 3)**:
- 减重10磅或5kg
- 根据MR10表现决定：
  - 完成10次 → 做10组×3次，组间休息60秒
  - 完成8-9次 → 做8组×3次，组间休息60秒
  - 完成7次 → 做5组×3次，组间休息60秒
  - **少于7次 → 跳过深蹲减量组，后续几周的1RM重量调低至少2.5%**

### 7.9 第2周动态调整逻辑

**关键点**: MR10的表现需要在训练执行时动态判断，影响后续训练

```
Day 1 深蹲MR10记录
├── 记录实际完成次数
├── ≥8次 → 正常进行加量组 (5组×3次)
└── <8次 → 最大重量减轻2.5%后继续加量组

Day 3 深蹲MR10记录 + 减量组决策
├── 记录实际完成次数
├── 10次 → 10组×3次
├── 8-9次 → 8组×3次
├── 7次 → 5组×3次
└── <7次 → 跳过减量组 + 提示用户调低后续1RM
```

**实现要点**:
1. 训练执行时需要记录MR10的实际次数
2. 根据次数动态展示后续组数
3. 如果<7次，弹出提示让用户确认是否调整后续1RM
4. 调整后的1RM影响第3-5周的训练重量

### 7.8 第6周设计需求

根据Excel原文，第6周有三个选项：

**选项1: 跳过**
- 用第5周完成的1-4次重量来预估新1RM
- 预估系数: 2次×1.03, 3次×1.06, 4次×1.09
- 直接开始下一轮周期

**选项2: 减载周**
- 用预估1RM开始下一轮
- 但在进入前做一周减载
- 重做第一周的训练，但不做最后一个上肢项

**选项3: 测试1RM**
- 用第6周来测定实际1RM
- 然后再减载或进入下一轮

|------|------|
